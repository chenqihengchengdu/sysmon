sysmon

Linux 终端下的 系统资源监控工具，按用户聚合 CPU 和内存使用。

pip install --user 无需 sudo 纯 Python

1. 安装 2. 自动记录 3. 实时柱状图 4. 单人详情 5. 复盘检测 6. 7 天走势 7. 命令速查 8. 常见问题

1. 安装

# 一行安装，无需 sudo
pip install --user git+https://github.com/chenqihengchengdu/sysmon.git

如果安装后提示 sysmon: command not found，把 ~/.local/bin 加入 PATH：

echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc

要求：Python 3.9+、psutil（自动安装）。仅读取 /proc，不需要 root。

2. 自动记录

历史趋势和复盘分析依赖定时记录。添加一条 cron 任务，每 5 分钟记录一次：

*/5 * * * * /path/to/sysmon --record

数据存储在 ~/.local/share/sysmon/sysmon.db (SQLite)，关机不丢失。

没有历史数据的话，走势图和复盘功能无法使用。建议安装完立刻加上 cron。

5. 复盘检测 — 找出资源波动元凶

核心功能

sysmon --review 扫描指定时间窗口内 所有用户、所有记录，找出 单次间隔内增长最快 的用户。

谁从空闲突然飙到 100% CPU？谁在 5 分钟内吃掉 10 GB 内存？一目了然。

============================ Review — Peak MEM Spike (last 12h) =============================
User                         Peak Spike (MEM%)
user1   ██████████████████████████████████████████████    12.4%  (14:35)
user2   ██████████████                                   3.1%  (09:10)
user3   ████                                            1.2%  (16:50)
使用场景命令
怀疑有人内存泄漏sysmon -review
怀疑有人 CPU 飙高sysmon -review --cpu
只看最近 3 小时sysmon -review --hour 3
只看最近 1 小时sysmon -review --hour 1
为什么这个功能重要？ 多用户服务器上，一个人的失控任务可能耗尽资源，影响所有人。 sysmon -review 能在用户抱怨之前告诉你 谁爆发了、爆发了多少、什么时间。

6.5 内存报警

设置阈值后，每分钟自动检查剩余内存。超标时通过 wall 弹窗通知所有在线用户，报警一次后自动停止。

sysmon -alert start 50G        # 剩余内存低于 50GB 报警
sysmon -alert start 20%        # 剩余内存低于 20% 报警
sysmon -alert start 50G --row 5  # 报警时 wall 重复 5 行（默认 9 行）
sysmon -alert stat              # 查看当前报警配置
sysmon -alert stop              # 停止报警

报警生效期间，所有 sysmon 命令都会在顶部显示红色 !!! MEMORY LOW !!! 横幅。

工作机制： sysmon -alert start 自动写入 crontab，每分钟执行一次静默检查。 触发后 wall 广播、自动停止（不刷屏）。用 --row N 控制 wall 消息重复行数。

8. 常见问题

Q: 安装后 sysmon 命令找不到？

pip install --user 把脚本装在 ~/.local/bin，该目录可能不在 PATH 中：

echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc

Q: 走势图 / 复盘没有数据？

需要先配置自动记录（cron）并等待数据积累。每 5 分钟一条，至少需要几条才能画图。

Q: 某个用户内存占用特别大（几百 GB）正常吗？

sysmon 统计的是 RSS（物理内存），不区分共享页。如果用户跑了大量并行任务或 fork 的子进程，汇总值可能较大。这个值反映的是"该用户进程总共占据的物理内存"。可结合 free -h 对比系统总内存。

Q: CPU 超过 100% 是什么意思？

已归一化到 0-100%（除以 CPU 核心数）。如果看到接近 100%，说明该用户几乎占满了所有核心。
